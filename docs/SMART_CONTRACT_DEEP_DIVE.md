# Smart Contract Deep Dive

## 1. Overview and Philosophy
The `main.compact` smart contract acts as the ultimate arbiter of truth for the ZERA protocol on the Midnight Network. It is written in Compact, a domain-specific language optimized for zero-knowledge circuits and privacy-preserving state transitions.

ZERA adheres to a strictly "Zero-Trust" architecture. The backend API is treated merely as a highly untrusted relayer. If the backend is compromised, it cannot forge asset ownership, steal assets, or mint counterfeits because every critical state mutation requires a mathematically verifiable zero-knowledge proof generated by the end-user's wallet.

## 2. Ledger State Architecture
The ledger is partitioned into public and shielded maps.

- `assetCount: Counter`: A monotonically increasing integer ensuring every asset receives a unique global identifier.
- `assets: Map<Integer, AssetRecord>`: Stores immutable data established at mint time. The `AssetRecord` includes the `assetHash`, the `metadataHash` (bridging the IPFS payload to the chain), and the timestamp.
- `commitments: Map<Bytes, Boolean>`: A critical security map. By storing the raw `assetHash` upon registration, the contract inherently rejects any future attempt to mint an identical asset, solving the widespread NFT duplication problem.
- `ownershipCommitments: Map<Integer, Bytes>`: The shielded ownership state. Instead of mapping `Asset ID => Wallet Address`, it maps `Asset ID => SHA256(Domain Separator + Public Key)`.

## 3. Circuit Logic and Execution Flows

### 3.1 `registerAsset`
**Purpose**: Securely initialize a new digital asset on the ledger.
**Execution Flow**:
1. The client generates an off-chain witness using their `creatorSecretKey`.
2. The circuit receives the `assetHash`, `metadataHash`, and the current network `timestamp`.
3. The circuit checks the `commitments` map. If the `assetHash` evaluates to `true`, the circuit aborts with a duplicate error.
4. The circuit derives the creator's public key from the secret key witness.
5. The asset is written to the `assets` map, the `assetHash` is flagged as `true` in `commitments`, and `assetCount` is incremented.

### 3.2 `assignOwnership`
**Purpose**: Bootstraps the ownership commitment immediately following `registerAsset`.
**Execution Flow**:
Because Compact explicitly prevents reading a value in the exact same block it is initialized, `assignOwnership` acts as the bridge. It accepts the newly created `assetId` and initializes the `ownershipCommitments` map with the creator's commitment hash. This readies the asset for secondary market trading.

### 3.3 `transferOwnership`
**Purpose**: Transfer an asset privately from the current owner to a buyer.
**Execution Flow**:
1. The circuit receives the `assetId` and the `newOwnerPublicKey` (provided by the buyer).
2. The seller provides their `ownerSecretKey` as a private witness.
3. The circuit hashes the seller's derived public key against the domain separator.
4. **Security Check**: The circuit strictly asserts that the generated hash exactly matches the hash currently stored in `ownershipCommitments[assetId]`. If it fails, the transaction is rejected instantly.
5. The circuit destroys the old commitment.
6. The circuit hashes the `newOwnerPublicKey` against the domain separator and writes the new commitment to the ledger.

### 3.4 `verifyOwnership`
**Purpose**: Allows third-party auditors (or the frontend UI) to mathematically verify ownership without mutating state.
**Execution Flow**:
1. Receives the `assetId` and the expected `publicKey`.
2. Hashes the `publicKey` locally.
3. Returns a boolean indicating if the local hash matches the on-chain commitment.

## 4. Threat Models and Mitigations

### 4.1 Double-Spend and Reentrancy
**Threat**: A user attempts to sell the same asset to two different buyers by broadcasting two simultaneous transactions.
**Mitigation**: The Midnight ledger processes transactions sequentially. The first transaction executes the `transferOwnership` circuit, successfully validating the owner's commitment and immediately replacing it with the buyer's commitment. When the second transaction executes, the circuit attempts to validate the seller's commitment again, but the ledger state has already mutated. The validation assertion fails, and the second transaction reverts.

### 4.2 Frontrunning and MEV
**Threat**: A malicious miner or indexer observes a high-value asset being minted in the mempool and attempts to submit a higher-gas transaction to mint the exact same `assetHash` under their own creator key.
**Mitigation**: The `assetHash` generated off-chain contains a cryptographic signature signed by the original creator. Even if a frontrunner duplicates the payload, the circuit derives the public key from the signature and ties the asset provenance to the true creator permanently. The frontrunner merely pays gas to mint an asset explicitly stamped with someone else's cryptographic identity.

### 4.3 Replay Attacks
**Threat**: An attacker takes an ownership commitment from a testnet and broadcasts it to the mainnet to seize control of an asset.
**Mitigation**: The ownership commitment generation utilizes strict domain separation: `commitment = persistentHash(["zera:production:mainnet:ownership", publicKey])`. Because the domain separator is explicitly baked into the circuit, cross-network and cross-contract replay attacks are mathematically impossible.
